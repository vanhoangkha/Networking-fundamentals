INTRODUCTION đến Mạng TỰ ĐỘNG HÓA WHY Mạng Tự động hóa - Previous versions của the CCNA focused trên the traditional model của managing / controlling networks - The current version focuses trên the traditional model as well, but CCNA candidates là expected đến have một basic understanding của various topics related đến Mạng Tự động hóa - In the traditional model, engineers manage devices one tại một time bởi connecting đến their CLI via SSH --- DOWNSIDES của CONFIGURING DEVICES ONE-BY-ONE - Typos và other small mistakes là common - It là time-consuming và very inefficient trong large-scale networks - It là difficult đến ensure that all devices ADHERE đến the organization’s STANDARD Cấu hình --- BENEFITS của Mạng Tự động hóa - Human Error (Typos, etc) is reduced - Networks become much more scalable và implemented trong một fraction của the time - New deployments - Mạng-wide changes - Khắc phục sự cố - Mạng-wide policy compliance có thể be assured - Standard configurations - Software versioning - The improved efficiency của Mạng operations reduces the OP-EX (operating expenses) of the Mạng. Each task requires fewer MAN-hours ``` There are various tools / methods that can be used to automate tasks in the network - SDN (Software-Defined Networking) - Ansible - Puppet - Python scripts - etc… ``` --- LOGICAL “PLANES” OF Mạng FUNCTIONS **What does một Router do?** - It forwards messages between networks bởi examining information trong the Layer 3 Header - It uses một Định tuyến Giao thức like OSPF đến share Tuyến đường information với other routers và build một Định tuyến table - It uses ARP đến build một ARP table, mapping IP Addresses đến MAC Addresses - It uses Syslog đến keep logs của events that occur - and MUCH more… **What does một Switch do?** - It forwards messages within một LAN bởi examining information trong the Layer 2 Header - It uses STP đến ensure there là no Layer 2 loops trong the Mạng - It builds một Địa chỉ MAC table bởi examining the Source Địa chỉ MAC của frames - It uses Syslog đến keep logs của events that occur - It allows một user đến connect đến it via SSH và manage it --- The various functions của Mạng devices có thể be logically divided up (categorized) into *PLANES* ``` - DATA PLANE - CONTROL PLANE - MANAGEMENT PLANE ``` - The operations của the Quản lý PLANE và the Kiểm soát PLANE là usually managed bởi the CPU - However, this là not desirable cho DATA PLANE operations because CPU processing là slow (relatively speaking) - Instead, a specialized hardware ASIC (Application-Specific Integrated Circuit) is used. - ASICs là chips built cho một specific purpose - Using một Switch, as một example: - When một Khung là received, the ASIC (not the CPU) is responsible cho the Chuyển mạch logic - The Địa chỉ MAC table là stored trong một kind của memory called TCAM (Ternary Content-Addressable Memory) - Another common name cho the Địa chỉ MAC table là CAM TABLE - The ASIC feeds the DESTINATION Địa chỉ MAC của the Khung into the TCAM which returns the matching Địa chỉ MAC table entry - The Khung là then forwarded out của the appropriate Thiết bị - Modern ROUTERS also use một similar hardware DATA PLANE: An ASIC designed cho forwarding logic, and tables store trong TCAM --- A SIMPLE SUMMARY: >- When một Thiết bị receives Kiểm soát / Quản lý traffic (destined cho itself), it sẽ be processed trong the CPU >- When một Thiết bị receives DATA traffic which nên pass through the Thiết bị, it là processed bởi the ASIC cho maximum speed --- DATA PLANE - All tasks involved trong forwarding USER DATA / TRAFFIC từ one Giao diện đến another là part của the DATA PLANE - A Router receives một message, looks cho the most specific matching Router trong its Định tuyến TABLE, and forwards it out của the appropriate Giao diện đến the next hop - It also de-encapsulates the original LAYER 2 Header, and re-encapsulates với một new Header destined cho the next hop’s Địa chỉ MAC - A Switch receives một message, looks tại the DESTINATION Địa chỉ MAC, and forwards it out của the appropriate Giao diện (or FLOODS it) - This includes functions like adding / removing 802.1q VLAN tags - NAT (changing the SRC / DST addresses before forwarding) is part của the DATA PLANE - Deciding đến forward / discard messages due đến ACL’s, Cổng-Bảo mật, etc. is part của the DATA PLANE - The DATA PLANE là also called the ‘FORWARDING PLANE’ ![image](https://github.com/psaumur/CCNA/assets/106411237/6a72186b-2956-45f6-8643-801caa2cb28e) --- Kiểm soát PLANE - How does một Thiết bị’s DATA PLANE make its forwarding decisions? - Định tuyến TABLE - Địa chỉ MAC table - ARP table - STP - etc… - Functions that build THESE tables (and other functions that influence the DATA PLANE) are part của the Kiểm soát PLANE - The Kiểm soát PLANE *controls* what the DATA PLANE does, Ví dụ bởi building the Router’s Định tuyến TABLE - The Kiểm soát PLANE performs *overhead* work - OSPF itself doesn’t forward user data packets, but it informs the DATA PLANE about HOW packets nên be forwarded - STP itself isn’t directly involved trong the process của forwarding FRAMES, but it informs the DATA PLANE about which INTERFACES nên và shouldn’t be used đến forward FRAMES - ARP messages aren’t user data but they là used đến build một ARP TABLE which là used trong the process của forwarding data ![image](https://github.com/psaumur/CCNA/assets/106411237/4c21b082-5d6e-4388-94c5-bebf33b50c8d) --- Quản lý PLANE - Like the Kiểm soát PLANE, the Quản lý PLANE performs overhead work - However, the Quản lý PLANE doesn’t directly affect the forwarding của messages trong the DATA PLANE - The MANAGMENT PLANE consists của PROTOCOLS that là used đến manage devices - SSH / Telnet : Used đến connect đến the CLI của một Thiết bị đến configure / manage it - SYSLOG : Used đến keep logs của events that occur trên the Thiết bị - SNMP : Used đến monitor the operations của the Thiết bị - NTP : Used đến maintain accurate time trên the Thiết bị ![image](https://github.com/psaumur/CCNA/assets/106411237/3cfffa40-f4cb-4042-8778-139605c2eb26) --- SOFTWARE-DEFINED NETWORKING (SDN) - SOFTWARE-DEFINED NETWORKING (SDN) is một approach đến networking that centralizes the Kiểm soát PLANE into một application called a *CONTROLLER* - SDN là also called SOFTWARE-DEFINED-Kiến trúc (SDA) or CONTROLLER-BASED NETWORKING - Traditional Kiểm soát PLANES use một distributed Kiến trúc - Ví dụ: - Each Router trong the Mạng runs OSPF và the ROUTERS share Định tuyến information và then calculate their preferred routes đến each destination - An SDN CONTROLLER centralized Kiểm soát PLANE functions like calculation routes - That là just một example và how much của the Kiểm soát PLANE là centralized varies greatly - The CONTROLLER có thể interact programmatically với the Mạng Thiết bị using APIs (Application Programming Giao diện) ![image](https://github.com/psaumur/CCNA/assets/106411237/05c4c5d9-5ba4-480c-9c13-72fa1f7937db) --- SOUTHBOUND Giao diện (SBI) - The SBI là used cho communications between the CONTROLLER và the Mạng DEVICES it controls - It typically consists của một COMMUNICATION Giao thức và API (Application Programming Giao diện) - APIs facilitate data exchanges between programs - DATA là exchanged between the CONTROLLER và the Mạng DEVICES - An API trên the Mạng DEVICES allows the CONTROLLER đến Truy cập information trên the DEVICES, Kiểm soát their DATA PLANE TABLES, etc. - Some examples của SBIs : - OpenFlow - Cisco OpFlex - Cisco OnePK (Open Mạng Environment Platform Kit) - NETCONF --- NORTHBOUND Giao diện (NBI) - Using the SBI, the CONTROLLER communicates với the managed DEVICES và gathers information about them: - The DEVICES trong the Mạng - The Cấu trúc mạng (how the DEVICES là connected together) - The available INTERFACES trên each Thiết bị - Their CONFIGURATIONS - The NORTHBOUND Giao diện (NBI) is what allows us to: - Interact với the CONTROLLER - Truy cập the DATA it gathers about the Mạng - Program the Mạng - Make changes đến the Mạng via the SBI - A REST API (Representational State Transfer) is used trên the controller as một Giao diện cho APPS đến interact với it - OSGi (Java Open Services Gateway Initiative) - Java based NBI API - DATA là sent trong một structured (*serialized*) format such as JSON hoặc XML - This makes it easier cho programs đến use the DATA ![image](https://github.com/psaumur/CCNA/assets/106411237/d980626f-f731-46a4-ba14-72c3d21f2fd3) --- Tự động hóa trong TRADITIONAL NETWORKS VS SDN - Networking tasks có thể be automated trong traditional Mạng architectures too: - SCRIPTS có thể be written (ie: using Python) to push commands đến many DEVICES tại once - Python với good use của REGULAR EXPRESSIONS có thể parse through “show” commands đến gather information about Mạng devices - However, the robust và centralized DATA collected bởi SDN CONTROLLERS greatly facilitates these functions - The CONTROLLER collects information about all DEVICES trong the Mạng - NORTHBOUND APIs allow APPS đến Truy cập information trong một format that là easy cho programs đến understand (ie: JSON và XML) - The centralized DATA facilitates Mạng-wide analytics - SDN Tools có thể provide the benefits của Tự động hóa without the requirement của third-party scripts và apps. - You don’t need expertise trong Tự động hóa đến make use của SDN Tools - However, APIs allow third-party applications đến interact với the CONTROLLER, which có thể be very powerful >💡 Although SDN và Tự động hóa aren’t the same thing, the SDN Kiến trúc greatly facilitates the Tự động hóa của various tasks trong the Mạng via the SDN CONTROLLER và APIs
